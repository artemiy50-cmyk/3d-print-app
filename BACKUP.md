# Правила и состав бэкапов

Документ описывает два канала резервного копирования данных приложения и интернет-магазина (ИМ): экспорт из веб-приложения и серверный скрипт `backup.py`.

---

## Сценарии использования

**Полный дамп (`FULL_DB.json` из `backup.py`)**  
Резерв «на чёрный день»: восстановление всей Realtime Database целиком. На практике применяется редко; хранится для аудита и аварий, когда недостаточно точечного отката.

**Пользовательский снимок (основной рабочий сценарий)**  

1. Файл **`user_{uid}.json`** с сервера по расписанию **или** JSON, скачанный пользователем из приложения («Скачать базу»).  
2. Корень JSON совпадает с содержимым **`users/{uid}/data`** (изделия, филаменты и т.д.) плюс служебные поля.  
3. Пользователь **входит в свой аккаунт** и загружает этот файл через **«Восстановить»**.  
4. Приложение записывает данные в облако так же, как при собственном экспорте: основная база в `data`, витрина и заказы — по ключу **`_storeBackup`**, подписка — в **`users/{uid}/subscription`** по ключу **`_subscription_backup`** (не внутрь `data`).

Таким образом, **per-user файл из `backup.py` рассчитан на передачу пользователю и восстановление через UI**, без ручной правки JSON. Нужны действия пользователя: подтверждение импорта и вход в аккаунт, иначе блок ИМ из файла в Firebase не применится (как и при экспорте из приложения).

## 1. Бэкап из приложения («Скачать базу»)

**Где:** главное приложение, пункт меню экспорта JSON.

**Что попадает в файл**

| Область | Путь в Firebase | Примечание |
|--------|------------------|------------|
| Основная база приложения | `users/{uid}/data` | Изделия, филаменты, настройки и т.д. — объект `db`, как в облаке. |
| Витрина | `users/{uid}/store` | Настройки магазина, в т.ч. поддомен и оформление. Узел копируется **целиком** — в бэкап входят поля SEO (`seoTitle`, `seoDescription`, `seoOgImage`, `seoNoindex`) и Яндекс (`yandexVerificationMeta`, `yandexMetricaSnippet`) без отдельного списка в коде. |
| Каталог | `users/{uid}/storeProducts` | Товары витрины. |
| Категории | `users/{uid}/storeCategories` | Дерево категорий. |
| Заказы ИМ | `storeOrders` | Только заказы с `ownerUid`, совпадающим с текущим пользователем. |
| Поддомен | `storesBySubdomain/{subdomain}` | Одна запись для поддомена из `store.subdomain`, если он задан. |

**Служебные поля в JSON**

- `_backupVersion: 2` — версия формата (при экспорте под залогиненным пользователем).
- `_storeBackup` — объект с данными ИМ; перед записью в `users/.../data` эти ключи из объекта импорта удаляются, чтобы не сохранялись внутри `data`.

**Ограничения**

- Экспорт **без входа** в аккаунт сохраняет только локальный объект приложения (`db`), **без** блока ИМ и без `_backupVersion`.
- Узел **`storeOrderSeq`** в файл **не входит** — это общий счётчик номеров заказов для всего проекта; его намеренно не копируют и не восстанавливают из файла приложения.
- Другие ветки под `users/{uid}/`, кроме перечисленных (например отдельные настройки профиля, если появятся), в этот экспорт **не включены**, пока логика экспорта их явно не добавит.

**Восстановление («Восстановить»)**

- В `users/{uid}/data` записывается содержимое бэкапа без служебных ключей (`_backupVersion`, `_storeBackup`, устаревший алиас `_store_backup`, `_subscription_backup`).
- Если в файле есть **`_subscription_backup`**, после сохранения `data` значение записывается в **`users/{uid}/subscription`** (не в объект приложения в `data`).
- Если в файле есть **`_storeBackup`** (или для совместимости со старыми файлами **`_store_backup`**) и пользователь **залогинен**, выполняется запись витрины, каталога, категорий, заказов владельца и соответствующей записи `storesBySubdomain`. У заказов принудительно выставляется `ownerUid` текущего пользователя. Заказы в файле могут быть **массивом** `{ id, ... }` (как в экспорте приложения и в `backup.py`) или **объектом** `{ orderId: данные }` — оба варианта поддерживаются.
- Если блок ИМ есть, а входа нет — `data` может обновиться только если сработало сохранение в облако; витрина и заказы из файла **не применяются** (показано сообщение в интерфейсе).

**Формат файла**

- Сохранение через **Blob** в браузере (без ограничения длины `data:` URL на больших JSON).

## 2. Серверный бэкап (`backup.py`)

**Назначение:** снимок Realtime Database по расписанию (отдельный репозиторий/CI).

**`FULL_DB.json`**

- Дамп корня `/` целиком: все `users`, `storeOrders`, `storesBySubdomain`, `storeOrderSeq` и любые другие узлы.
- Используется для полного восстановления БД или расследования состояния на дату снимка.

**`user_{uid}.json` (per-user, для выдачи пользователю и «Восстановить»)**

- Копия **`users/{uid}/data`** (`deepcopy`), в том же виде, что ожидает импорт в приложении (корень = поля `db`).
- При наличии подписки: **`_subscription_backup`** — копия `users/{uid}/subscription` (при импорте уходит в `users/{uid}/subscription`, не в `data`).
- При наличии данных ИМ или заказов: **`_storeBackup`** (как в `script.js`) с полями:
  - `store`, `storeProducts`, `storeCategories` — как в Firebase;
  - `storeOrders` — **массив** объектов `{ "id": "<ключ Firebase>", ...поля заказа }`, как при «Скачать базу»;
  - `storesBySubdomain` — объект `{ subdomain: значение }` для поддомена из `store`, если запись есть в корне.

**Замечания**

- Аутентификация Firebase: файл **`firebase-key.json`** по пути `FIREBASE_KEY_PATH` **или** переменная **`FIREBASE_SERVICE_ACCOUNT`** (JSON строкой). Содержимое env **не сохраняется** в файл в репозитории — иначе при каждом запуске без локального ключа в клоне появлялся бы `firebase-key.json`.
- **`storeOrderSeq`** есть только в полном дампе, в per-user файл **не дублируется**.
- Устаревший ключ **`_store_backup`** (snake_case) в старых снимках приложение при импорте всё ещё понимает; новые файлы `backup.py` пишут только **`_storeBackup`**.

**Удаление старых снимков в git-репозитории (не в `backup.py`)**  

В отдельном приватном репозитории с бэкапами в **GitHub Actions** (`backup.yml`) после `backup.py` часто идёт шаг **очистки каталога `backups/`**: например, `find backups -mindepth 1 -maxdepth 1 -type d -mtime +180 -exec rm -rf {} +` — удаляются **подпапки**, у которых на раннере **mtime** старше **180 суток**. Это ограничивает рост репозитория; логика живёт в **workflow**, а не в Python-скрипте. Условие завязано на **время изменения папки** в рабочей копии, а не на парсинг имени `YYYY-MM-DD_…`; при смене политики хранения меняют порог (`+180`) или сам шаг в YAML.

## 3. Согласованность и восстановление

- Для **полного** отката состояния БД на момент времени ориентируйтесь на **`FULL_DB.json`** и процедуры восстановления Firebase (импорт/merge — по вашей политике).
- Для **отката одного пользователя на дату** используйте **`user_{uid}.json`** или экспорт из приложения; передайте файл владельцу и попросите **«Восстановить»** под своим логином.
- При ручном merge из полного дампа учитывайте **`storeOrderSeq`** и чужие заказы в `storeOrders`.
- После изменений схемы данных (новые узлы под `users/{uid}/` или в корне) этот документ и код бэкапов стоит обновить в одном изменении.
