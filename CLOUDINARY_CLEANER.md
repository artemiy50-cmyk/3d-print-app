# Очистка Cloudinary и сверка с Firebase (`cleaner.py`)

## Назначение

Скрипт раз в месяц (или по расписанию в CI) решает две задачи:

1. **Белый список URL** — собирает все ссылки на `cloudinary.com`, которые считаются «нужными», потому что встречаются в актуальной базе Realtime Database.
2. **Очистка хранилища** — сравнивает каталог ресурсов в Cloudinary с этим списком и **удаляет** файлы, на которые в БД никто не ссылается, чтобы облако не забивалось устаревшими загрузками.

Дополнительно (режим «лечения»): если в БД есть ссылка на Cloudinary, а файла в аккаунте уже нет, скрипт может **обнулить или подчистить** это поле в Firebase, чтобы интерфейс не показывал битые картинки.

По сравнению с ранней версией скрипта добавлен **учёт раздела ИМ** (настройки витрины и товары каталога): ссылки из этих узлов попадают в белый список и при лечении могут правиться, иначе картинки магазина ошибочно считались бы «лишними» в Cloudinary.

## Где в БД ищутся ссылки

| Источник | Путь | Поля |
|----------|------|------|
| Изделия | `users/{uid}/data/products` | `imageUrl`, вложения `fileUrls[].url` |
| Настройки витрины | `users/{uid}/store` | `banner`, `logo`, `seoOgImage` (если это URL Cloudinary) |
| Товары ИМ | `users/{uid}/storeProducts` | `imageUrl`, массив `imageUrls[]` |

Категории магазина (`storeCategories`) в текущей схеме **не** хранят картинки — в обход не берутся.

Заказы (`storeOrders`) в сохранённом виде содержат позиции без встраиваемых URL превью — отдельный обход заказов для whitelist **не** делается (при появлении снимков URL в заказах логику нужно расширить).

Учитываются только строки, в которых подстрока **`cloudinary.com`** (как в клиентском коде приложения).

## Порядок работы скрипта

1. **Инициализация** — Cloudinary API (ключ/секрет) и Firebase Admin (service account + `databaseURL`).
2. **Список ресурсов Cloudinary** — постранично (`max_results=500`, `next_cursor`), в память попадают `secure_url`, `url` и `public_id`.
3. **Проход по ветке** `users/*` (каждый пользователь):
   - для каждого пользователя читаются `data.products`, `store`, `storeProducts`;
   - каждая валидная ссылка Cloudinary, для которой найден соответствующий ресурс в облаке, попадает в множество **активных URL**;
   - при включённом лечении битые ссылки обнуляются/вырезаются и пишутся обратно точечными `update` в Firebase.
4. **Кандидаты на удаление** — ресурсы Cloudinary, чьи `secure_url` и `url` ни разу не вошли в активный список.
5. **Удаление** — пакетами по 100 `public_id` через `cloudinary.api.delete_resources` (если не отключено переменной окружения).

## Переменные окружения

| Переменная | Назначение |
|------------|------------|
| `CLOUDINARY_CLOUD_NAME` | Имя облака (по умолчанию как в приложении, можно переопределить). |
| `CLOUDINARY_API_KEY` | API Key Cloudinary (**обязательно**). |
| `CLOUDINARY_API_SECRET` | API Secret (**обязательно**). |
| `FIREBASE_KEY_PATH` | Путь к JSON ключа (по умолчанию `firebase-key.json`). |
| `FIREBASE_DB_URL` | URL Realtime Database. |
| `FIREBASE_SERVICE_ACCOUNT` | Если файла ключа нет — содержимое JSON сервисного аккаунта (как в `backup.py`). |
| `FIX_BROKEN_LINKS` | `true` / `false` — править ли битые ссылки в БД (по умолчанию `true`). |
| `DELETE_ORPHAN_CLOUDINARY` | `true` / `false` — удалять ли «лишние» файлы в Cloudinary (по умолчанию `true`). Удобно для пробного прогона: только отчёт без удаления. |

## Риски и рекомендации

- Удаление в Cloudinary **необратимо**. Имеет смысл первый прогон с `DELETE_ORPHAN_CLOUDINARY=false` и проверить число «кандидатов».
- Если какие-то URL используются **вне** перечисленных узлов (другой сайт, ручные вставки, будущие поля), такие файлы могут ошибочно попасть под удаление — расширяйте в скрипте **список учитываемых узлов** БД.
- Сервисный ключ Firebase и секреты Cloudinary **нельзя** коммитить в git; в CI — только секреты репозитория или секретный менеджер.

## Файлы рядом со скриптом в вашем util-репозитории

### `requirements.txt`

Список зависимостей Python для окружения, где крутятся утилиты (тот же набор подходит и для `backup.py`, и для `cleaner.py`):

- **`firebase-admin`** — доступ к Realtime Database от имени сервисного аккаунта.
- **`cloudinary`** — официальный SDK для Admin API (список и удаление ресурсов).

Установка: `pip install -r requirements.txt`.

### `d-print-app-3655b-firebase-adminsdk-*.json`

Это **ключ сервисного аккаунта Google** для проекта Firebase (`service_account`). Внутри — `private_key` и учётные данные робота; с ним можно читать и писать БД в пределах правил и ролей IAM.

**Назначение локально/в CI:** подставлять путь к этому файлу в `FIREBASE_KEY_PATH` или один раз выставить `FIREBASE_SERVICE_ACCOUNT` содержимым файла — как в вашем исходном `cleaner` и в `backup.py`.

**Безопасность:** файл с реальным ключом **не должен** попадать в публичный репозиторий. Если он когда-либо был закоммичен или отправлен в чат — **перевыпустите ключ** в [Google Cloud Console](https://console.cloud.google.com/) (IAM → Service Accounts → ключи) и обновите секреты в CI. В основном репозитории приложения такой JSON **не хранится** — только `cleaner.py` / `backup.py` и документация.
