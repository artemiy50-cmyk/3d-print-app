# Вывод магазина в прод на поддомене (GitHub Pages + sync)

Пошаговая настройка **одного** магазина на **третьем уровне**: `your-shop.example.com`  
(ниже везде плейсхолдеры — подставьте свой поддомен, домен второго уровня и GitHub-логин.)

**Идея:** код Store в **одном** основном репозитории (`store.html`, `store.js`, `style.css`). Для **каждого** поддомена — **отдельный** небольшой репозиторий под GitHub Pages; workflow **Store Sync** копирует файлы и создаёт `CNAME`.

### Простыми словами

- **Пустой deploy-репо — это нормально.** Вы создаёте на GitHub пустой репозиторий под магазин (например `store-luckyart`). В него **сам** положит файлы workflow из основного репо при первом успешном **Store Sync**. То есть дело **не** в том, что «некуда клонировать»: место как раз это пустое репо.
- **Ошибка вроде `Invalid username or token` или код 128 при clone/push** почти всегда значит: секрет **`STORE_SYNC_TOKEN`** в основном репо не подходит — токен неверный, истёк, или (если токен *fine-grained*) в его настройках **не указан доступ** к нужному репозиторию `store-…`. Пока это не исправить, Sync не заработает **ни для одного** нового магазина.

### Новый магазин: порядок по шагам (как делать по времени)

Делайте **по номерам сверху вниз**. Подробности — в разделах «Шаг 0» … «Шаг 8» ниже.

| Порядок | Что сделать |
|--------|-------------|
| 1 | **Шаг 0:** поддомен в Manager (для LuckyArt — `luckyart`), витрина сохранена |
| 2 | **Шаг 1:** на GitHub создать **пустой** репозиторий `store-{поддомен}` (например `store-luckyart`) |
| 3 | **Шаг 3:** секрет **`STORE_SYNC_TOKEN`** в основном репо; PAT лучше **classic** с **`repo`** (см. шаг 3 ниже). При **fine-grained** каждый новый `store-…` нужно вручную добавлять в список репозиториев токена. |
| 4 | **Шаг 4:** в `store-deploy-config.json` добавить **ещё одну** пару `repo` + `subdomain` в массив `deployRepos` (старые записи не удалять). Push в `main`. |
| 5 | **Шаг 5:** в **основном** репо **Actions → Store Sync to Deploy Repos → Run workflow** (ветка `main`). В deploy-репо ничего не запускать. После успеха — ветка `main` и файлы в `store-…`. |
| 6 | **Шаг 2:** в deploy-репо **Settings → Pages**: источник **Deploy from a branch**, ветка **`main`**, папка **`/(root)`**, Save. *(Раньше не получится, если в репо ещё не было коммитов — обычно после шага 5 уже есть `main`.)* |
| 7 | **Шаг 6:** у регистратора DNS: CNAME поддомен магазина → `ваш-логин.github.io` |
| 8 | **Шаг 7–8:** в Pages указать свой домен третьего уровня, включить HTTPS при необходимости, открыть сайт в браузере |

## Что можно держать в публичном репозитории

| Обычно **не** считается секретом | **Нельзя** светить |
|----------------------------------|-------------------|
| URL сайтов, поддомены, имена репозиториев GitHub | Пароли, приватные ключи, **Personal Access Token** (PAT), содержимое `firebase-key.json` |
| Публичные параметры **Firebase** из клиентского кода (`apiKey`, `databaseURL`, `projectId` и т.д.) — они и так видны в браузере; доступ режется **правилами БД** и настройками в консоли Firebase | Сервисный аккаунт Google, секреты Cloudinary в репо (держите в GitHub Secrets / локально) |
| `store-deploy-config.json` — раскрывает **имена** репозиториев (`owner/repo`), не пароли | Секрет `STORE_SYNC_TOKEN` хранится только в **Settings → Secrets** репозитория, в git не попадает |

Если не хотите афишировать связку «какой GitHub-аккаунт какие магазины ведёт» — делайте основной или deploy-репозитории **приватными** (учитывайте лимиты GitHub и правила Pages для приватных репо).

## Что должно быть заранее

- Основной репозиторий с приложением, ветка **`main`**.
- Файл `.github/workflows/store-sync.yml` в основном репо.
- Домен второго уровня (например **`example.com`**) с DNS у регистратора (**reg.ru** или другой).
- Значения для подстановки:
  - **`YOUR_SHOP`** — поддомен **строчными** латиницей, как в URL: `your-shop` → `your-shop.example.com`.
  - **`YOUR_DOMAIN`** — `example.com` (без `www` для записи CNAME поддомена).
  - **`YOUR_GITHUB_USER`** — логин или организация на GitHub.
  - **`YOUR_PAGES_HOST`** — обычно `YOUR_GITHUB_USER.github.io` (цель CNAME в DNS).

## Шаг 0. Поддомен в Firebase и в Manager

1. Войдите в **Manager** (ваш прод-URL приложения).
2. Раздел магазина / витрины.
3. Укажите **поддомен** ровно **`YOUR_SHOP`** (как в будущем URL и в `storesBySubdomain`).
4. Заполните витрину (отображаемое имя магазина, товары и т.д.), **сохраните**.

Проверка (опционально): в Realtime Database есть `storesBySubdomain/YOUR_SHOP` → ваш `ownerUid`, и данные `users/{uid}/store`, `storeProducts` и т.д.

## Шаг 1. Репозиторий под GitHub Pages для этого магазина

1. GitHub → **New repository**.
2. **Owner:** `YOUR_GITHUB_USER`.
3. **Имя:** например **`store-YOUR_SHOP`** (соглашение `store-{поддомен}`).
4. Для бесплатного GitHub Pages на **своём** домене чаще нужен **Public** репозиторий — уточните [актуальные правила GitHub](https://docs.github.com/pages).
5. Без README / .gitignore / license (пустой репозиторий).
6. **Create repository.**

Полное имя: **`YOUR_GITHUB_USER/store-YOUR_SHOP`**.

## Порядок: Pages не настраивают сразу после пустого репо

Если репозиторий создан **без** README и **без** коммитов, в нём **нет ветки `main`**. Тогда в **Settings → Pages** в выпадающем списке **Branch** будет только **None** — выбрать **`main`** и папку **`/(root)`** нельзя, пока не появится хотя бы один коммит.

**Рекомендуемый порядок по времени:** **шаг 1** (создать пустой репо) → **шаги 3, 4, 5** (токен, `store-deploy-config.json`, **Store Sync**) → затем **шаг 2** (Pages: `main` + `/(root)`), потом **6–8**. В документе «Шаг 2» идёт выше по файлу, но настраивать Pages нужно **после** появления ветки `main` (обычно после Sync).

**Альтернатива без ожидания Sync:** на странице пустого репо нажмите **Add file → Create new file**, введите имя `README.md`, любой текст, **Commit** — появится **`main`**. Тогда можно зайти в Pages и выбрать **main** / **`/(root)`** (позже Sync всё равно перезапишет содержимое).

## Шаг 2. Включить GitHub Pages в deploy-репо

**Делайте после того, как в репозитории уже есть ветка `main`** (обычно — после успешного **шага 5**).

1. Репозиторий **`store-YOUR_SHOP`** → **Settings** → **Pages** (слева в меню настроек).
2. Блок **Build and deployment** → **Source:** **Deploy from a branch** (не GitHub Actions).
3. **Branch:** первый выпадающий список — выберите **`main`** (не *None*). Второй список — папка: **`/(root)`** — это **корень** репозитория (не `/docs`).
4. Нажмите **Save**.

Первые минуты сайт может отдавать 404 или старую заглушку — нормально, пока не обновится билд Pages.

## Шаг 3. Токен (PAT) и секрет `STORE_SYNC_TOKEN`

### Что считать «основным» репозиторием

**Основной репо** — тот, где лежат **`store.html`**, **`store.js`**, **`style.css`**, файл **`store-deploy-config.json`** и workflow **`.github/workflows/store-sync.yml`**. Actions запускаются **в нём** и оттуда же читают секрет.

У вас это репозиторий **`artemiy50-cmyk/3d-print-app`** (или ваш `логин/имя-репо` с тем же содержимым). Репозитории вида **`store-luckyart`** — это **deploy-репо** (отдельные); секрет **`STORE_SYNC_TOKEN`** в них **не** создаётся, он хранится **только** в основном репо.

### Часть A. Создать или обновить Personal Access Token (PAT)

Это настройки **аккаунта GitHub**, не вкладка Settings внутри репозитория.

#### Что выбрать: classic или fine-grained

**Для Store Sync однозначно удобнее — Tokens (classic)** с правом **`repo`**.

| | **Classic (`repo`)** | **Fine-grained** |
|---|----------------------|------------------|
| **Что влияет** | Один и тот же токен может пушить во **все** ваши репозитории (в т.ч. любые будущие `store-…`) без правок токена | Доступ только к **явно отмеченным** репозиториям |
| **Новый магазин** | Обычно **ничего** не менять в PAT, только добавить репо в `store-deploy-config.json` | После создания каждого нового `store-…` зайти в токен и **добавить этот репо** в список; иначе снова *Invalid username or token* |
| **Безопасность** | Шире по правам (все репо аккаунта под `repo`) | Уже по охвату, но больше ручной работы и риск забыть репо |

Итого: если не нужна жёсткая минимизация прав — нажимайте в левом меню **Tokens (classic)** → **Generate new token (classic)** и включайте **`repo`**. Страницу **Fine-grained** для этого сценария можно **не** использовать.

#### Пошагово: classic (рекомендуется)

1. Аватар → **Settings** (профиль) → **Developer settings**.
2. **Personal access tokens** → **Tokens (classic)** → **Generate new token (classic)**.
3. **Note:** любое имя (например `store-sync-3d-print-app`).
4. **Expiration:** по желанию; по истечении нужно создать новый токен и обновить секрет **`STORE_SYNC_TOKEN`**.
5. Отметьте **`repo`** (целиком чекбокс раздела *repo*).
6. **Generate token** → **сразу скопируйте** строку токена (потом не покажут).

#### Если всё же fine-grained

1. **Personal access tokens** → **Fine-grained tokens** → **Generate new token**.
2. **Repository access** → **Only select repositories** → отметьте **все** deploy-репо (`store-test-shop`, `store-luckyart`, …).
3. **Permissions** → **Repository permissions** → **Contents: Read and write**, **Metadata: Read**.
4. Создайте токен и скопируйте значение.

### Часть B. Положить токен в секрет репозитория `3d-print-app`

Здесь вы уже внутри **репозитория** с приложением.

1. Откройте на GitHub репозиторий **`3d-print-app`** (вкладка с кодом — не важно).
2. Верхняя панель → **Settings** (настройки **этого** репозитория).
3. В **левом** меню найдите блок **Security** → **Secrets and variables** → нажмите стрелку раскрытия → пункт **Actions**.
4. Если секрета ещё нет: **New repository secret**. Имя в точности: **`STORE_SYNC_TOKEN`**, в поле **Secret** вставьте **целиком** скопированный PAT (без пробелов в начале/конце). Сохраните.
5. Если секрет уже есть, но вы сменили токен или расширили права fine-grained: откройте **`STORE_SYNC_TOKEN`** → **Update secret** → вставьте новое значение (при смене classic/fine-grained проще заново создать токен и обновить секрет).

В **git** и в **pull request** токен **не** коммитить — только в этом месте GitHub Secrets.

### Когда добавляете новый магазин (новое `store-…`)

- **Classic PAT с `repo`** — обычно **ничего** менять в токене не нужно, только убедиться, что секрет в `3d-print-app` актуален.
- **Fine-grained** — зайдите в **часть A**, отредактируйте токен и **добавьте новый** репозиторий `store-…` в список; при необходимости обновите секрет, если выпустили новый PAT.

## Шаг 4. Конфиг `store-deploy-config.json` в основном репо

Добавьте объект в массив `deployRepos`:

```json
{
  "deployRepos": [
    {
      "repo": "YOUR_GITHUB_USER/store-YOUR_SHOP",
      "subdomain": "YOUR_SHOP"
    }
  ],
  "githubPagesBase": "YOUR_GITHUB_USER.github.io"
}
```

Несколько магазинов — несколько элементов в `deployRepos`.

Поле **`githubPagesBase`:** должно быть **`ВЛАДЕЛЕЦ.github.io`**, где **владелец** — та же строка, что в начале каждого `repo` до символа `/` (ваш логин или организация). Пример: `repo` = `artemiy50-cmyk/store-luckyart` → **`githubPagesBase`:** `artemiy50-cmyk.github.io`. Это тот хост, который вы указываете **в значении DNS-записи CNAME** у регистратора (куда «смотрит» поддомен магазина).

**Не путать с файлом `CNAME` в deploy-репо:** после Sync там лежит **ваш** домен третьего уровня (`YOUR_SHOP.YOUR_DOMAIN`), например `luckyart.my-3d-print.ru`. Это верно и **не** должно совпадать с `githubPagesBase`. Схема: браузер → ваш домен → DNS говорит «иди на `username.github.io`» → GitHub Pages отдаёт сайт из репо.

### Как проверить, что всё согласовано

1. **JSON** — в `store-deploy-config.json` сравните:
   - во всех `repo` один и тот же владелец перед `/`;
   - `githubPagesBase` = этот владелец + `.github.io` (без `https://`, без `/`).

2. **DNS у регистратора** (шаг 6) — откройте зону **`YOUR_DOMAIN`**, найдите **CNAME** для поддомена магазина (**имя** обычно `YOUR_SHOP` или полное `YOUR_SHOP.YOUR_DOMAIN` — как принято в вашей панели). Поле **значение / каноническое имя** должно быть **точно** `YOUR_GITHUB_USER.github.io` (тот же текст, что в `githubPagesBase`). Если там другой логин или опечатка — сайт не откроется на вашем домене.

3. **Профиль GitHub** — если сомневаетесь в логине: откройте `https://github.com/YOUR_GITHUB_USER` — имя в URL совпадает с префиксом в `repo` и с поддоменом `github.io`.

4. **После сохранения DNS (опционально)** — с ПК: `nslookup -type=CNAME YOUR_SHOP.YOUR_DOMAIN` (подставьте полное имя, например `luckyart.my-3d-print.ru`). В ответе должен быть канонический хост вида **`something.github.io`**.

**Что можно прислать для проверки:** скрин **таблицы DNS** с видимой строкой CNAME (имя поддомена + значение) и скрин или фрагмент **`store-deploy-config.json`** с `repo` и `githubPagesBase`. Достаточно сравнить глазами: значение CNAME = строка `githubPagesBase`.

Commit + push в **`main`**.

## Шаг 5. Первый запуск синхронизации

### Откуда запускать

**Только из основного репозитория** (там же, где `store-deploy-config.json` и секрет **`STORE_SYNC_TOKEN`**): например **`YOUR_GITHUB_USER/3d-print-app`**. В deploy-репо (`store-YOUR_SHOP`) workflow **не** настраивается и **ничего** вручную запускать не нужно — туда Actions **сами отправляют** файлы после успешного прогона в основном репо.

### Ручной запуск

1. На GitHub откройте **основной** репозиторий.
2. Вкладка **Actions**.
3. В левом списке выберите workflow **Store Sync to Deploy Repos**.
4. Справа кнопка **Run workflow** → выберите ветку **`main`** → **Run workflow**.

### Автозапуск

При **push в `main`** основного репо, если изменились любые из файлов:

- `store.html`, `store.js`, `style.css`
- `store-deploy-config.json`
- `.github/workflows/store-sync.yml`

— тот же workflow стартует **сам**, отдельно нажимать **Run workflow** не обязательно.

### После запуска

Убедитесь, что job **успешен**. В **`store-YOUR_SHOP`** на **`main`** должны появиться: `index.html`, `store.js`, `style.css`, при наличии — `favicon.ico`, файл **`CNAME`** с одной строкой:

```text
YOUR_SHOP.YOUR_DOMAIN
```

(без `https://`)

## Шаг 6. DNS у регистратора (пример reg.ru)

1. Личный кабинет → ваш домен **`YOUR_DOMAIN`** → управление DNS / зоной.
2. Новая запись:
   - **Тип:** `CNAME`.
   - **Имя / поддомен:** `YOUR_SHOP` (только первая метка, без `.YOUR_DOMAIN`).
   - **Значение:** **`YOUR_GITHUB_USER.github.io`** (без `https://`, без `/`).
   - **TTL:** по умолчанию или 3600.

3. Сохранить. Подождать распространения DNS (от минут до нескольких часов).

Интерфейс reg.ru может называть поля чуть иначе; суть — CNAME с поддомена магазина на `username.github.io`.

## Шаг 7. Кастомный домен в GitHub Pages

1. Репозиторий **`store-YOUR_SHOP`** → **Settings** → **Pages**.
2. **Custom domain:** `YOUR_SHOP.YOUR_DOMAIN` → **Save**. Дождитесь **DNS check successful** (зелёная галочка).
3. **Enforce HTTPS** GitHub включит **только после выпуска TLS-сертификата** для этого домена (Let’s Encrypt). Пока сертификат не готов, пункт будет **неактивен** с текстом вроде *certificate has not yet been issued* — это нормально, не значит, что домен введён неправильно. Обычно проходит **от минут до суток**; обновляйте страницу настроек позже и включите **Enforce HTTPS**, когда галочка станет доступна.
4. До появления сертификата сайт может открываться по **`http://`**, а **`https://`** и антивирусы могут ругаться на имя сертификата — ожидаемо до готовности HTTPS.
5. Если **DNS successful**, но сертификат **не появляется дольше суток**, проверьте у регистратора записи **CAA** для **`YOUR_DOMAIN`**: они не должны запрещать выпуск сертификатов для Let’s Encrypt (см. [Troubleshooting GitHub Pages custom domains](https://docs.github.com/pages/configuring-a-custom-domain-for-your-github-pages-site/troubleshooting-custom-domains-and-github-pages)).

## Шаг 8. Проверка

Откройте **`https://YOUR_SHOP.YOUR_DOMAIN`** (или временно **`http://`**, пока в шаге 7 недоступен **Enforce HTTPS**). Должен открыться Store; данные из Firebase по поддомену **`YOUR_SHOP`**. При ошибках — поддомен в Manager, `storesBySubdomain`, консоль браузера (F12).

## SEO и ссылки на товары (в Manager)

В разделе **Мой магазин** → вкладка **SEO и соцсети** задаются поля для поиска и соцсетей (заголовок страницы, meta description, загрузка картинки **og:image** в Cloudinary, опция **noindex**, **метатег подтверждения домена для Яндекс Метрики**, поле **кода счётчика Яндекс Метрики** для витрины). Сохранение — та же кнопка **Сохранить**, что и для настроек витрины (доступна и на вкладке SEO). Ссылка на загруженное изображение хранится в Firebase; скрипт **`cleaner.py`** учитывает поле **`seoOgImage`** в белом списке (как баннер и логотип), чтобы файл не удалялся при ежемесячной уборке.

На опубликованной витрине при открытии карточки товара в адрес добавляется параметр **`?tovar=…`**: слаг из **транслитерации названия** товара (латиница, дефисы; при совпадении имён — суффиксы **-2**, **-3**…). Ссылку можно скопировать из браузера и отправить клиенту. Локальная проверка: **`store.html?store=ВАШ_ПОДДОМЕН`** (без отдельного домена для теста).

**Вебмастер (добавить сайт в поиск Яндекса):**

1. Откройте [https://webmaster.yandex.ru/](https://webmaster.yandex.ru/) и войдите под **Яндекс ID** (если аккаунта нет — зарегистрируйтесь на [yandex.ru](https://yandex.ru)).
2. Нажмите **Добавить сайт** и укажите полный адрес витрины, например **`https://ВАШ_ПОДДОМЕН.ваш-домен.ru/`** (тот же URL, по которому открывается магазин после GitHub Pages и DNS).
3. Подтвердите права на сайт одним из способов, который предложит сервис: **HTML-файл** в корне репозитория витрины, **meta-тег** в шаблоне (или через поля SEO в Manager, если сервис даёт универсальный метатег), **DNS-запись** у регистратора домена и т.п. Без подтверждения Вебмастер не привяжет ресурс к вашему аккаунту.

**Нужно ли вручную добавлять sitemap в Вебмастер?** Обычно **нет**: на витрине в **`robots.txt`** уже есть строка **`Sitemap:`** с полным URL **`https://ваш-поддомен.домен/sitemap.xml`** (файлы создаются при **Store Sync** / **Refresh Store Sitemaps**, см. абзац **Sitemap** ниже). Робот Яндекса может подхватить карту сайта оттуда без ваших действий. Если всё же хотите явно указать карту в кабинете: в Вебмастере откройте **сайт** → раздел **Индексирование** → **Файлы Sitemap** → в поле ввода укажите тот же URL **`https://ваш-поддомен.домен/sitemap.xml`** → **Добавить**. Это **необязательный** шаг; на скорость появления страниц в выдаче он не даёт гарантий. Индексация в любом случае идёт с задержкой (дни и дольше).

**Метрика (аналитика посещений витрины):**

1. Откройте [https://metrika.yandex.ru/](https://metrika.yandex.ru/) под тем же или другим Яндекс-аккаунтом.
2. **Счётчики** → **Добавить счётчик**. Укажите **адрес сайта** (URL витрины), имя счётчика, при необходимости цели и согласия с условиями.
3. Если Метрика просит **подтвердить владение сайтом** через **HTML** / **метатег**, скопируйте предложенный тег **`meta name="yandex-verification"`** (или только длинное значение из атрибута **content**). В **Manager**: **Мой магазин** → **SEO и соцсети** → поле **«Подтверждение домена для Яндекс Метрики через метатег»** → вставьте → **Сохранить**. На витрине в **`<head>`** появится этот метатег (`yandexVerificationMeta` в Firebase; **Store Sync не обязателен**).
4. На шаге с кодом счётчика нажмите **Скопировать код** — это обычно один или два тега **`<script>`** и иногда блок **`<noscript>`** с картинкой.
5. В **Manager**: **Мой магазин** → **SEO и соцсети** → поле **«Код счётчика Яндекс Метрики»** → вставьте скопированный фрагмент целиком → **Сохранить**. Код сохраняется в Firebase (`yandexMetricaSnippet`) и подставляется на публичной витрине при загрузке страницы (**отдельный Store Sync для этого не обязателен** — данные читаются из базы).
6. В интерфейсе Метрики откройте **Отчёты** и проверьте, что после визита на витрину появляются визиты (иногда с задержкой в несколько минут).

**Sitemap:** файл **`sitemap.xml`** (главная + URL с **`?tovar=`** для активных товаров) и **`robots.txt`** со строкой **`Sitemap:`** создаётся скриптом при **Store Sync** (когда вы пушите код витрины). Новые товары в Manager **не** запускают этот workflow сами по себе. Чтобы карта сайта не отставала от каталога, в основном репо есть отдельный workflow **Refresh Store Sitemaps**: он **раз в сутки по расписанию** (UTC) пересобирает только `sitemap.xml` и `robots.txt` во всех deploy-репо из `store-deploy-config.json` и пушит коммит, если что-то изменилось. Тот же секрет **`STORE_SYNC_TOKEN`**. Вручную: **Actions → Refresh Store Sitemaps → Run workflow**. Расписание меняется в **`.github/workflows/store-sitemap-refresh.yml`** (cron). Если в Manager включён **noindex**, в **`robots.txt`** пишется **`Disallow: /`**, sitemap остаётся пустым. Указать sitemap в Search Console — по желанию.

### Что уже сделано по SEO (кратко)

| Область | Реализация |
|---------|------------|
| Manager | Вкладка **SEO и соцсети**: title, description, изображение для превью, **noindex**, **метатег yandex-verification**, код **Яндекс Метрики** |
| Витрина (`store.js`) | Подстановка **title**, **meta description**, **canonical**, **Open Graph**, **Twitter card**, JSON-LD (**WebSite** / **Product** в модалке), **`?tovar=`** + транслит; **`yandexVerificationMeta`** в **head**; вставка **`yandexMetricaSnippet`** в **head** / **body** |
| Deploy | **Store Sync** + ежедневный **Refresh Store Sitemaps** → **`sitemap.xml`**, **`robots.txt`** |
| Облако | **seoOgImage** (и **logo**) в белом списке **`cleaner.py`** |

Позиции в выдаче Google/Яндекса **не гарантируются** — зависят от конкуренции, контента и поведения пользователей. Реализация закрывает **техническую** базу: корректные мета-теги, расшариваемые ссылки на товары, карта сайта, опция скрытия от индекса.

### Как проверить, что всё настроено и работает

1. **Вкладка и сохранение** — **Мой магазин → SEO и соцсети** заполнены, нажата **Сохранить** (зелёный тост). После изменений витрины — **Store Sync** в GitHub, чтобы на сайт ушли актуальные **`store.js`** / **`index.html`**.

2. **Главная витрины в браузере** — откройте прод-URL (HTTPS). **Просмотр кода страницы** (не «Элементы»): в `<head>` должны быть **`meta name="description"`**, **`og:title`**, **`og:description`**, при наличии картинки — **`og:image`**, **`link rel="canonical"`**. Заголовок вкладки = заданный **title** (или из настроек магазина). Если включён **noindex** — есть **`meta name="robots" content="noindex"`**.

3. **Карточка товара** — откройте товар, в адресе появится **`?tovar=...`**. Снова **просмотр кода**: **title** и **description** должны отражать товар; **og:type** для товара; внизу `<head>` может быть блок **JSON-LD** с типом **Product**.

4. **Sitemap и robots** — в браузере: **`https://ВАШ_ПОДДОМЕН.домен/sitemap.xml`** и **`/robots.txt`**. В sitemap есть главная и строки с **`tovar`**; в robots — ссылка **Sitemap:** (если не **noindex**). После добавления товаров подождать суточный **Refresh Store Sitemaps** или запустить workflow вручную.

5. **Внешние проверки (по желанию)** — после стабильного HTTPS: [Facebook Sharing Debugger](https://developers.facebook.com/tools/debug/) или аналоги для **VK/Telegram** — вставить URL главной и URL с **`?tovar=`** и убедиться, что превью, заголовок и картинка подтягиваются. [Google Rich Results Test](https://search.google.com/test/rich-results) — проверка разметки страницы. **Google Search Console** / **Яндекс.Вебмастер** — добавить ресурс, при необходимости указать URL **sitemap.xml**; индексация идёт с задержкой в дни–недели.

6. **Локально** — **`store.html?store=поддомен`**: мета-теги и **tovar** работают так же; поисковики на localhost обычно не ходят.

## Обновления кода магазина

Изменения в **`store.html`**, **`store.js`**, **`style.css`** в основном репо → push в **`main`** → sync обновит **все** репо из `deployRepos`. Ручной запуск — только в **основном** репо: **Actions** → **Store Sync to Deploy Repos** → **Run workflow** (ветка `main`).
